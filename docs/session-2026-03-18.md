### 5. AIO Data Population on CSV Upload (`app/page.tsx`)

When a user uploads and processes a CSV file, each generated AIO is now saved to the `aio_data` table in addition to `information_objects`:

- `aio_name` = `"<filename> - Row N"`
- `element_1`…`element_50` = each `[Key.Value]` bracket string from the AIO line (up to 50)

Parsing uses `parseAioLine()` which extracts all `[…]` bracket tokens via regex.

---

### 6. Duplicate CSV Detection (`app/page.tsx`)

Before processing uploaded files, the app checks the backend for existing CSV records:

- Queries `GET /api/io?type=CSV&source_system=csv-converter&limit=500`
- Compares filenames against `context.source_object_id` of existing records
- Shows a toast error for each duplicate: _"filename.csv is already in the database. Please pick a different file."_
- Skips duplicates; if all files are duplicates, aborts processing entirely
- Check is silently skipped when the backend is offline

## Root Cause Fixed

The backend was missing the `roles`, `aio_data`, and `hsl_data` endpoints entirely, and the database migration to create those tables had never been written. This caused all CRUD operations in the System Management screen to fail silently ("failed to save"). Both the migration and the endpoints were added in commit `52e8424`.

## Architecture Notes

- Migration files in `infophysics_impl_grade/migrations/` are run on container startup by `start.sh` via `glob.glob('migrations/*.sql')` (sorted). All migrations are idempotent (`CREATE TABLE IF NOT EXISTS`, `ON CONFLICT DO NOTHING`).
- The backend uses `psycopg` (sync) with a context-manager `db()` helper for connection management.
- Element arrays are stored as 50 (or 100) individual columns and reconstructed as Python lists on read.
- The frontend API client (`lib/api-client.ts`) uses `safeFetch()` which returns `null` on any backend error, keeping the UI resilient when the backend is unavailable.
